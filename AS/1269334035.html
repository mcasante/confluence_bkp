Problema

Time de pricer precisa validar API de precificação, e para isso, precisa dos dados transacionais dos clientes + dados do google analytics

Já temos todos os dados transacionais de todos os clientes que receberam desembolsos da a55.

Logo, a validação do modelo com eles é mais simples, já que o unico dado pendente é o de GA

O problema é que precisamos de uma forma para incentivar esses clientes já desembolsados a compartilhar os seus dados de GA

Solução / Hipótese

  1. Chamariz dentro da área logada (possivelmente no hub de crédito). Clientes do prime que toparem compartilhar os seus dados de google ads terão carencia no custo de emissão de boletos da plataforma

Proposta de valor única

Modelo de precificação automatizado, com base em dados de GA (precisa somente ser validado).

Não há outro banco/fintech que consegue precificar somente com esse tipo de dado.

Vantagem competitiva

Esse modelo nos traz a vantagem de oferecer ao cliente uma experiencia de contratação de crédito que não depende da analise de dados financeiros, cadastro de contas e upload de documentos.

Além dessa menor burocratização, esse modelo de pricer conseguiria dar um retorno real-time para nossos clientes.

Seguimento de clientes

Clientes da base da a55 (já desembolsados), que utilizem o Google Analytics (para que modelo seja validado)

Métricas-chave

Conseguir 50% dos clientes desembolsados e que possuem conta de Analytics com conta do GA integrada na plataforma, para que modelo possa ser validado

Canais

LP externa a plataforma, a ser enviada por e-mail (sugestão)

Estrutura de custo

Utilizaremos as APIs de integração já validadas e utilizadas na esteira de crédito, logo o custo seria de: 1. alocação de um front-end por um sprint, 2. Alocação de um Product Designer para prototipação da solução por um sprint 3. custo que a a55 assumirá da carencia que será dada aos clientes que emitem boletos na plataforma, como incentivo para compartilhamento dos dados.

Custo total = 4.500 (Front) + 5.000 (UX) + Incentivo (dado sendo coletado)

Fontes de receita

TAC dos desembolsos cuja analise foi feita usando nova API de pricer

Que problema(s) estamos resolvendo?

Temos um novo modelo de precificação, que precisa ser validado.

Para que a validação seja feita, precisamos de clientes que tenham nos fornecido tanto dados transacionais (extratos bancários/adquirentes), quanto realizado opti-in de suas contas do GA.

Por quê agora?

Porque a v1 da nova API de precificação acabou de ser entregue, e ela precisa ser validada antes de ser colocada no ar.

Sem ela, ainda dependeremos de analises manuais realizadas pelo time de Crédito, que demoram até 2 dias úteis para serem realizadas. Com ela no ar, esse tempo será reduzido para minutos

Além disso, essa api baseada no GA resolverá um de nossos grandes gargalos no processo de obtenção de crédito, que é obrigar os clientes a cadastrar suas contas bancárias (processo manual).

Objetivos estratégicos

-Atingir 25% dos desembolsos realizados para ads e digital totalmente tech-touch (16 desembolsos)


-Atingir 100MM em desembolsos para SaaS e Ecommerce

-Reduzir o tempo para que um lead em SQL vá para etapa de análise de 7 para 5 dias corridos


O que define o sucesso dessa iniciativa?

O sucesso da iniciativa é conseguirmos 50% dos clientes que receberam desembolso e que possuem conta de analytics com sua conta de GA integrada em nossa base de dados.

Como mensuramos os resultados

-Mensurar quantos clientes que receberam desembolso possuem conta de GA

-Desse montante, precisamos que 50% tenham vinculado suas contas em nossa base
