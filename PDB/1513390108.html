Processo de Bugs : Processo de resolução de Bugs

Created by Tech A55, last modified by Eduardo Cavalcanti on Feb 09, 2022

Contexto

Atualmente a identificação e gestão de bugs acontece centralizada em poucas pessoas e apenas por grupos do Google Chat. Para melhorar o gerenciamento e o senso de urgência de todos do time em relação ao impacto do bug no dia a dia da empresa, vamos utilizar do apoio do Jira para fazer esse meio de campo entre todas as pessoas envolvidas.

Proposta de Fluxo

  1. Inclusão de Bugs no Hubspot

  2. Automação ticket Hubspot <> Jira

  3. Gestão do PO para resolver bug junto ao seu squad

    1. PO analisar card no board do de bugs

    2. Alocar um desenvolvedor responsável

    3. Vincular cards do bug e de resolução da tarefa (sub-tarefas)

    4. Fazer transições do card no board de bug / Comunicar stakeholders (CX/Clientes)

    5. Priorizar bugs com as tarefas em desenvolvimento

  4. Validação final com quem trouxe o bug

  5. Processo de fechamento do ticket por CX/Sales

Como escrevemos bugs

Documentar bem um bug é tão importante quanto encontrá-lo. Um bug bem documentado é o meio caminho do trabalho que alguém de produto e de tecnologia precisam para priorizar e resolver o problema da maneira mais adequada.

Uma boa documentação precisa transmitir o impacto que o problema encontrado está trazendo para a operação bem como dar o máximo de contexto para que o desenvolvedor descubra qual é o problema.

Padrão de documentação

Ao criar um card de bug, tentamos preencher as seguintes informações:

  1. Título

  2. Cenário

  3. Resultado esperado

  4. Resultado ocorrido

  5. Fluxo do problema

  6. Anexos do fluxo

Título

Uma frase concisa dizendo qual foi o problema encontrado.

Cenário

Informações para contextualizar o problema. Queremos entender quem está sendo afetado, em que etapa do fluxo, se o problema acontece sempre ou apenas algumas vezes, se acontece para todos os usuários, etc… Pontos importantes a serem trazidos:

QUEM ESTÁ SENDO IMPACTADO?

  1. Clientes internos (CX/Sales/Formalização)

  2. Clientes externos (Parceiros/Usuários/Sacados)

    1. Relevância do cliente impactado (Relação/Tamanho do Deal)

EM QUE SISTEMA ESTÁ DANDO O ERRO?

  1. Plataforma

  2. Midgard

  3. HubSpot

ETAPA DO FLUXO ONDE O PROBLEMA OCORREU

  1. Landing pages

  2. Aquisição de dados

  3. Precificação

  4. Formalização

  5. Desembolso

  6. Acompanhamento do crédito

IMPACTO DO PROBLEMA

  1. Alto: o usuário não é capaz de concluir a ação desejada ou tem acesso a uma informação errada crítica para a sua operação, ex. data de vencimento errada

  2. Médio: o usuário é capaz de concluir a ação desejada mas é difícil pensar em um fluxo alternativo ou o fluxo alternativo está disponível apenas para alguns perfis de usuário

  3. Baixo: desconforto na jornada ou informação pouco relevante para o usuário (ex. texto com erro gramatical)

VOLUME DO PROBLEMA

  1. Alto: todos os clientes de um mesmo perfil são impactados pelo problema (ex. todos os usuários do fluxo de aquisição, todos os parceiros)

  2. Médio: grupo específico de clientes é impacto

  3. Baixo: apenas um cliente é impactadoImpacto

FREQUÊNCIA DO PROBLEMA

  1. Alto: estimativa de mais de 10% das vezes em que um usuário passa pelo fluxo, ocorre o problema

  2. Médio: estimativa entre 10% e 2% das vezes em que um usuário passa pelo fluxo, ocorre o problema

  3. Baixo: estimativa de menos de 2% das vezes em que um usuário passa pelo fluxo, ocorre o problema

Resultado esperado

Descrição do comportamento esperado pelo cliente. Pense assim: O que deveria ter acontecido quando o usuário clicou em um botão? O que deveria ter acontecido x dias após determinado momento?

Exemplo: Ao clicar no menu Propostas (Crédito), o botão Nova Proposta deveria aparecer na listagem de propostas e ser clicável.

Resultado ocorrido

Descrição do comportamento ocorrido durante o uso pelo cliente.

Exemplo: Ao clicar no menu Propostas (Crédito), o botão Nova Proposta aparecia, mas não era possível clicar nele.

Fluxo do problema

Descrição do passo a passo até a ocorrência do problema. A descrição pode ser feita como uma lista numérica, para facilitar a escrita e o entendimento.

Exemplo:

  1. Entrei no Midgard com meu usuário normal

  2. Na interface principal > Cliquei no menu Propostas (Crédito)

  3. Na listagem de propostas > Botão Nova Proposta desabilitado

Anexos do fluxo

Sempre que possível, ajuda muito que os produteiros e desenvolvedores vejam o estado da tela onde o problema está acontecendo. Se quiser ajudar mais ainda, pode circular a área onde o problema foi identificado (como na imagem abaixo - ou melhor caso tenha talentos artísticos! 😅).

Requisitos

Identificação do bug

  • Explicar contexto do bug

  • Quem foi impactado pelo bug / Como foi identificado

  • Comportamento esperado do serviço

Fluxo de trabalho

  1. PO analisar card no board do de bugs

  2. Alocar um desenvolvedor responsável

  3. Vincular cards do bug e de resolução da tarefa (sub-tarefas)

  4. Fazer transições do card no board de bug / Comunicar stakeholders (CX/Clientes)

  5. Priorizar bugs com as tarefas em desenvolvimento

Priorização

  • Quem está sendo impactado

    • Clientes internos (CX/Sales/Formalização)

    • Clientes externos (Parceiros/Usuários/Sacados)

    • Relevância do cliente impactado (Relação/Tamanho do Deal)

  • Tipo de impacto no fluxo

    • Desconforto na jornada, mas segue no fluxo

    • Fluxo travado, mas com alternativa de desbloqueio manual

    • Fluxo travado com necessidade de análise

  • Volume

    • Cliente único impactado

    • Grupo específico de clientes

    • Todos os clientes da plataforma

  • Estágio da jornada

    • Estágio inicial

    • Esteira de formalização

    • Esperando desembolso

    • Linha já contratada

Pessoas envolvidas

  • Time de CX/Sales (Primeiro contato e ponte com o cliente)

    • Alimenta Hubspot com cards novos

    • Acompanha transição do card

    • Comunica cliente do processo da resolução do bug

  • Squad de produto (POs)

    • Coordenar identificação, priorização e resolução do problema

    • Comunicação transparente entre time de desenvolvimento e CX/Sales
